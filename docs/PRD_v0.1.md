## 1. 产品概述

### 1.1 背景

Cat_Sys 是一个面向猫舍（猫咪繁育机构）的多租户 SaaS CRM 系统，帮助猫舍管理猫咪信息、客户关系、销售流程和健康记录。系统采用微服务架构，支持商家多分店运营。

### 1.2 产品定位

为中小型猫舍提供一站式的数字化运营管理工具，以 SaaS 模式交付，商家注册即用、数据完全隔离、功能模块可按需配置。

### 1.3 核心目标

- 覆盖猫舍日常运营的核心流程（猫咪管理 → 客户跟进 → 销售转化 → 健康追踪）
- 支持多分店连锁经营，员工可跨分店协作
- 数据隔离安全可靠，商家间互不可见
- 功能模块弹性配置，不同商家按需启用

## 2. 用户角色

| 角色 | 描述 | 权限范围 |
|------|------|----------|
| 超级管理员 (Super Admin) | 平台开发者，管理所有商家和功能模块 | 全平台所有操作 |
| 商家主理人 (Merchant Owner) | 猫舍法人，注册即开通 | 所属商家全部权限，含创建/删除分店、添加/禁用员工 |
| 店长 (Manager) | 分店管理者 | 本分店全部数据查看 + 日常运营管理，不可删除分店、禁用员工 |
| 销售 (Sales) | 负责客户跟进、销售转化、合同签署 | 可登记录入、编辑猫咪/客户/销售数据、修改售价；不可做管理操作 |
| 护理员 (Care Staff) | 负责猫咪日常照护、健康记录 | 可登记录入、编辑猫咪/健康数据、修改售价；不可做管理操作 |

## 3. 功能模块清单

### 3.1 基础模块（默认强制开通）

| 模块编码 | 模块名称 | 核心功能 |
|---------|---------|---------|
| `cat_mgmt` | 猫咪管理 | 猫咪增删改查、基本信息（品种/毛色/出生日期/父母/照片）、状态管理（在售/已售/自留/退役/去世）、猫咪列表搜索筛选 |
| `customer_mgmt` | 客户管理 | 客户增删改查、联系方式（手机/微信/邮箱）、偏好记录（偏好品种/颜色/预算）、客户来源、跟进记录、状态管理（潜在/意向/已成交/已流失） |
| `branch_mgmt` | 分店管理 | 分店增删改查、分店信息（名称/地址/联系方式/营业状态）、分店列表 |
| `employee_mgmt` | 员工管理 | 员工账号创建/禁用/启用、员工角色分配（店长/销售/护理员）、员工调店、密码重置、员工列表 |
| `dashboard` | 基础仪表盘 | 本分店概览（猫咪总数/客户数/待办任务）、近期事件提醒 |
| `account` | 账号与权限 | 登录/登出、修改密码、切换分店、个人信息编辑 |

### 3.2 高级模块（超级管理员可配置开关）

| 模块编码 | 模块名称 | 核心功能 |
|---------|---------|---------|
| `sales` | 销售管理 | 猫咪预订登记、定金管理（收取/退还/转单）、合同签署管理、尾款追踪、销售统计 |
| `health_pro` | 健康管理 Pro | 疫苗计划与记录、驱虫日历与提醒、就医/手术记录、体检报告、健康提醒（下次驱虫/疫苗日期自动计算） |
| `contract` | 合同与证书 | 合同模板管理、电子合同签署、血统证书信息管理、证书打印（预留） |
| `finance` | 财务报表 | 收支流水记录（收入/支出）、分类统计（猫粮/医疗/用品等）、分店盈亏对比、月度/年度报表 |
| `bulk_ops` | 批量导入导出 | 猫咪信息 Excel 批量导入、客户信息 Excel 批量导入、数据导出（CSV/Excel） |
| `api_access` | API 接口 | 对外数据接口、第三方平台对接（预留） |
| `auto_followup` | 客户自动跟进 | 跟进任务自动分配、客户生日提醒、购猫回访提醒、沉睡客户自动提醒 |

## 4. 业务流程

### 4.1 商家注册到上线流程

用户注册 → 填写商家信息 → 注册即开通（模式A）→ 商家主理人账号生效 → 创建分店 → 添加员工 → 开始运营

### 4.2 猫咪管理流程

新增猫咪（填写基本信息+上传照片）→ 分配所属分店 → 设置状态（在售）→ 客户感兴趣 → 转入销售管理流程 → 售出 → 更新状态（已售）→ 记录销售信息

### 4.3 客户跟进到成交流程

新增客户 → 记录偏好 → 关联意向猫咪 → 跟进记录（电话/微信/到店）→ 预订 → 收取定金 → 签订合同 → 支付尾款 → 交接猫咪 → 客户转为已成交

### 4.4 员工跨分店协作流程

员工登录 → 默认进入上次工作分店 → 可通过顶部下拉切换分店 → 切换后上下文变更为目标分店 → 所有操作限定在目标分店范围内

### 4.5 操作审计规则

- 店长、销售、护理员的所有写入操作（增/删/改）均记录审计日志
- 前端展示「最后操作人」「最后操作时间」两个字段，来源于审计日志
- 超级管理员操作同样记录，不可删除

## 5. 非功能需求

### 5.1 数据隔离

- 商家间数据完全隔离：所有业务表包含 `merchant_id` 字段
- API 层强制校验 `merchant_id` 与当前登录用户匹配
- 超级管理员可跨商家查看（操作需记录审计日志）

### 5.2 安全性

- 密码加密存储（bcrypt）
- JWT Token 鉴权，30 分钟过期
- 登录失败 5 次锁定 15 分钟
- 所有敏感操作记录审计日志

### 5.3 性能要求

- 常规列表查询 < 200ms
- 单个服务 ≥ 100 QPS
- 支持 MySQL 读写分离（预留）

### 5.4 可用性

- 核心服务要求 99.9%
- 支持 Docker 容器化部署

---

## 6. 配置管理要求

- 所有环境特定配置（域名、端口、数据库连接、Redis 连接、JWT 密钥）统一通过 .env 配置文件管理
- api-gateway 对外端口固定为 9002，Docker 映射 9002:9002
- 外部访问路径前缀 /cat 可配置，便于换域名或迁移服务器

## 7. 约束条件

- 不使用多语言
- 不使用短信通知（v0.1 暂不实现）
- 不使用繁育管理模块（v0.1 暂不实现）
- 不实现套餐定价体系（v0.1 暂不实现）
- 不引入消息队列（v0.1，服务间同步 HTTP 通信）
- 基础模块永久强制开通，超级管理员只可开关高级模块
